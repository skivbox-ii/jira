# User Activity Hard-Open Diagnostics — Design

Дата: 2026-04-01

## Цель

Показать сырую активность в календаре и детализации дня без downstream-фильтрации по выбранным пользователям и без визуального схлопывания автора, чтобы коммиты и действия разных людей не терялись в общей ленте.

## Проблема

Сейчас цепочка после загрузки Jira-данных делает несколько дополнительных сужений:

- `repo-data-processor.js` повторно фильтрует repo-события по выбранным пользователям через сопоставление `name/displayName/key/accountId`.
- `unified-calendar.js` режет автора до первого слова через `surname(...)`, из-за чего разные люди могут выглядеть как один.
- `daily-detail.js` повторно фильтрует действия по `selectedUsers` и пытается раскладывать авторов по колонкам; неоднозначные случаи уводятся в `unmatched`.

Из-за этого исходные события не теряются на этапе запроса, но могут стать неразличимыми или исчезнуть на этапе представления.

## Решение

Для диагностического прохода сделать `hard-open` поведением по умолчанию в downstream-ветке календаря:

- оставить текущий scope загрузки Jira-данных: выбранные пользователи по-прежнему определяют, какие Jira worklog/changelog/comment данные загружаются;
- убрать повторную фильтрацию repo-событий по выбранным пользователям после dev-status загрузки;
- перестать пытаться повторно сопоставлять автора события с одним из `selectedUsers` в детализации дня;
- показывать автора действий полным видимым лейблом вместо сокращения до одного слова там, где это мешает диагностике.

## Что меняется

### `ujg-user-activity-modules/repo-data-processor.js`

- Аргумент `selectedUser` сохраняется для совместимости сигнатуры, но перестаёт ограничивать выдачу.
- Repo-события проходят только по факту наличия валидного timestamp и попадания в выбранный диапазон дат.
- Нормализация автора (`author`, `authorName`, `authorMeta`) сохраняется как есть.

### `ujg-user-activity-modules/unified-calendar.js`

- Repo-линии календаря перестают сокращать автора через `surname(...)`.
- При необходимости тот же принцип применяется к action-линиям Jira-блока, если именно сокращение делает людей неразличимыми.
- Пользовательские day chips остаются частью текущего Jira-summary, а не становятся repo-агрегацией.

### `ujg-user-activity-modules/daily-detail.js`

- Убирается downstream-фильтрация действий через `filterActionsBySelectedUsers(...)`.
- Убирается логика `unmatched` как обязательный путь для неоднозначных авторов.
- Team view строится по фактическим авторам действий дня, а не по заранее выбранному списку пользователей.
- User filter в day detail либо удаляется, либо становится неактивным; предпочтение — удалить, чтобы не оставлять ложное ощущение фильтрации.

## Что не меняется

- `api.js` и объединение Jira issues/details по ключу внутри одной загрузки.
- `request-cache.js`.
- `repo-api.js` и merge `repository/pullrequest` dev-status payloads.
- Выбор пользователей в header и последовательная загрузка Jira-данных по этим пользователям.

Это важно: транспортный дедуп и кеширование не считаются причиной текущей коллизии и не должны добавлять шум в диагностический режим.

## Поток данных после изменения

1. Пользователь выбирает `selectedUsers`.
2. `rendering.loadData()` по-прежнему грузит Jira-данные только для этих пользователей.
3. `repoApi.fetchRepoActivityForIssues()` по-прежнему грузит общий dev-status по найденным задачам.
4. `repoDataProcessor.processRepoActivity()` больше не отбрасывает события по автору и возвращает все repo-события в диапазоне.
5. `rendering.mergeRepoItemsIntoProcessed()` переносит repo-события в `processed.dayMap[date].repoItems`.
6. `unified-calendar` и `daily-detail` рендерят эти действия без повторного user-match и без сокращения автора до неразличимого ярлыка.

## Риски

- В календаре и детализации появятся repo-события людей, которые не входят в текущий `selectedUsers`, если они работали по тем же задачам в том же диапазоне. Для диагностики это ожидаемое поведение.
- UI станет плотнее: полные имена и отсутствие downstream-фильтрации увеличат объём контента в ячейках дня.
- Summary cards и user chips останутся Jira-ориентированными и не станут источником истины для repo-авторов. Это допустимо в рамках диагностического режима.

## Проверка

- В repo-календаре должны одновременно отображаться события нескольких авторов по одному дню без схлопывания в одно имя.
- В day detail не должно быть потери действий из-за повторного user-match.
- Сценарий с одним пользователем должен продолжать работать, но теперь без скрытия соседних repo-авторов по тем же задачам.
