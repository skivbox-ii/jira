# Timesheet: дизайн режима по разработчикам

**Дата:** 2026-03-06

**Статус:** approved

## Цель

Сделать `ujg-timesheet` понятным и пригодным для работы на больших командах: корректно показывать данные по разработчикам, явно отображать дату в ячейке, не терять множественные списания внутри дня и заменить неудобный мультиселект пользователей на searchable dropdown.

## Подтверждённый scope

1. Переименовать чекбокс `Отдельные календари` в `По разработчикам`.
2. Если режим `По разработчикам` включён и фильтр пользователей пустой, строить отдельный календарь для каждого найденного автора.
3. Если режим `По разработчикам` включён и часть пользователей выбрана вручную, строить отдельные календари только по выбранным.
4. Если режим `По разработчикам` выключен, оставлять один общий календарь.
5. В ячейке дня показывать дату в формате `dd.mm`, а в `title` давать полную дату.
6. Если по задаче за день есть несколько списаний, не схлопывать их без деталей: карточка задачи остаётся одна, но внутри отображается список отдельных worklog-записей.
7. Список пользователей заменить на компактный dropdown с поиском, прокруткой, чекбоксами, действиями `Все` и `Сбросить`.
8. Сохранить текущую совместимость с `state.selectedUsers`, URL hash и сохранёнными группами.

## Технические решения

### 1. Правильная модель данных для фильтрации по разработчику

Текущая модель хранит в записи задачи только агрегированные `seconds`, `authors` и `comments`. Этого недостаточно для режима `По разработчикам`, потому что при фильтрации по одному автору задача всё равно несёт суммарное время всех авторов за день.

Подтверждённое решение:

- В `_ujgCommon.loadWorklogsForDay()` сохранить массив `worklogs`.
- Каждый элемент `worklogs` должен содержать как минимум:
  - `authorId`
  - `authorName`
  - `seconds`
  - `comment`
  - `started`
- Поля `seconds`, `authors` и `comments` остаются как агрегаты для общего режима, но итог для конкретного пользователя должен пересчитываться из `worklogs`.

### 2. Проекция задачи под выбранных пользователей

Фильтрация больше не должна ограничиваться проверкой `item.authors`. Вместо этого нужна проекция записи задачи под активный набор пользователей:

- если фильтра нет, задача возвращается целиком;
- если фильтр есть, из `worklogs` остаются только нужные записи;
- `seconds`, `comments` и `authors` пересчитываются по оставшимся `worklogs`;
- если после фильтрации `worklogs` пустой, задача исключается из результата.

Это решение делает корректными:

- отдельные календари по разработчикам;
- общий календарь с фильтром по нескольким пользователям;
- отображение комментариев и сумм без двойного счёта.

### 3. Рендер карточки задачи

Карточка задачи сохраняет компактный верхний блок:

- ключ задачи;
- сумма по задаче за день;
- статус;
- estimate.

Ниже добавляется список отдельных списаний за день:

- автор;
- длительность;
- короткий комментарий, если он есть.

Итог дня и итог задачи остаются агрегированными, чтобы календарь не потерял читаемость.

### 4. Режим `По разработчикам`

Источник списка календарей:

- если `state.selectedUsers.length > 0`, использовать выбранных пользователей;
- иначе использовать всех пользователей из `state.users`, найденных в загруженных данных.

Таким образом чекбокс становится самодостаточным и не требует предварительного ручного выбора.

### 5. Новый фильтр пользователей

Вместо `<select multiple>` используется кастомный control:

- кнопка-резюме c количеством выбранных;
- popup/dropdown;
- поле поиска;
- список чекбоксов с прокруткой;
- кнопки `Все` и `Сбросить`.

Источник данных не меняется: `state.selectedUsers` остаётся основным состоянием, чтобы не ломать URL-параметры и сохранённые группы.

## Ограничения

- Без моков и без переписывания гаджета на другой стек.
- Минимально инвазивные изменения в текущий AMD/jQuery-код.
- Без изменения существующего формата сохранённых групп в `localStorage`.

## Критерии приёмки

1. Чекбокс `По разработчикам` без ручного выбора пользователей показывает отдельные календари по всем найденным авторам.
2. Календарь конкретного разработчика показывает только его списания и его суммы.
3. В каждой ячейке видно `dd.mm`, а не только число месяца.
4. При нескольких списаниях по одной задаче внутри дня видно каждую запись отдельно.
5. При большом числе пользователей фильтр позволяет искать, скроллить и массово выбирать пользователей.
