# User Activity Worklog Lag — Design

Дата: 2026-04-01

## Цель

Показать в User Activity, за какой день был списан worklog, явно отмечать несвоевременное внесение и агрегировать отставание по пользователям:

- в строках календаря и детализации дня;
- в верхней таблице пользователей;
- в недельной мини-таблице внутри календаря.

## Проблема

Сейчас User Activity показывает только `worklog.started`, поэтому в UI видно день, на который отнесён worklog, но не видно, когда пользователь реально внёс это списание в Jira.

Из-за этого по строке вида `16:06 ... Worklog 1ч` нельзя понять:

- это списание за текущий день или за вчера;
- было ли оно внесено вовремя;
- насколько пользователь накопил отставание за неделю.

## Контракт расчёта

Для каждого worklog используются два момента времени:

- `workedDay`: календарный день из `worklog.started`;
- `loggedAt`: момент первого внесения worklog, то есть `worklog.created`.

Отставание считается только если `loggedAt` попадает не в тот же календарный день, что и `workedDay`.

Если worklog внесён день в день, отставание равно `0`, даже если он был занесён вечером.

### Формула

1. `workedDayBoundary = начало следующего календарного дня после workedDay в 00:00`
2. `delayHoursRaw = max(0, (loggedAt - workedDayBoundary) / 3600000)`
3. `lagScoreHours = delayHoursRaw * spentHours / 24`

Где:

- `delayHoursRaw` показывает фактическую сырьевую задержку в часах;
- `lagScoreHours` показывает вклад конкретного worklog в суммарное отставание пользователя.

### Пример

Если:

- `workedDay = 2026-04-25`
- `loggedAt = 2026-04-26 08:00`
- `spentHours = 4`

то:

- `delayHoursRaw = 8ч`
- `lagScoreHours = 8 * 4 / 24 = 1.33ч`

Именно `1.33ч` участвует в сумме отставания по пользователю.

## Источник данных

Дополнительный API-запрос не нужен: `api.js` уже получает `/rest/api/2/issue/{key}/worklog`, а Jira worklog обычно содержит `started`, `created`, `updated`.

Изменение нужно только в том, как эти поля сохраняются и используются в `data-processor.js`.

## Изменения в модели данных

Для каждого worklog в `data-processor.js` нужно сохранить:

- `started`
- `created`
- `loggedAt`
- `workedDayKey`
- `isLate`
- `lagDurationHoursRaw`
- `lagScoreHours`

Эти же поля должны без потерь проходить через `processMultiUserData(...)` в:

- `dayMap[date].allWorklogs`
- `dayMap[date].users[user].worklogs`
- `issueMap[key].worklogs`

## Отображение в UI

## Строка worklog

В `daily-detail.js` и в Jira-блоке `unified-calendar.js` worklog должен явно показывать:

- за какой день он списан: `за 25.04`;
- когда был внесён: `внесено 26.04 08:00`;
- если внесён поздно: `отставание 8ч`.

Для позднего списания строка получает заметную warning-подсветку.

Важно: в строке показывается человеческое описание задержки (`8ч`, `1д 16ч`), а в агрегатах используется формула `lagScoreHours`.

## Верхняя таблица пользователей

В `summary-cards.js` в таблицу пользователей добавляется новая колонка `Отставание`.

В этой колонке показывается сумма `lagScoreHours` по всем worklog пользователя в выбранном периоде.

Текущее поведение таблицы по числу пользователей не меняется:

- если пользователей больше одного, таблица показывается;
- если выбран один пользователь, дополнительная верхняя таблица не появляется.

## Недельная мини-таблица в календаре

В `unified-calendar.js` для каждой недельной строки календаря добавляется компактный блок с итогом отставания за эту неделю:

- `ФИО`
- `Суммарное отставание`

Мини-таблица строится из видимых worklog текущей недели и должна уважать render-time user filter:

- при включённых всех пользователях видны все выбранные dashboard users;
- при выключении пользователя из календарного фильтра его недельное отставание исчезает из мини-таблицы той же перерисовкой.

Рекомендуемое место вывода: внутри правой weekly summary-ячейки `Σ`, под недельной суммой часов, чтобы не ломать сетку календаря.

## Общее правило видимости

Отставание считается только для тех worklog, которые уже попали в текущую выборку User Activity:

- `workedDay` находится внутри выбранного периода;
- worklog принадлежит выбранному пользователю;
- запись реально присутствует в загруженных данных.

Это значит, что дашборд не делает отдельного retrospective-поиска worklog за пределами текущего окна.

## Out of Scope

В эту задачу не входят:

- пересчёт по рабочим дням или `8ч` вместо формулы `* / 24`;
- игнорирование выходных;
- использование `worklog.updated` вместо `worklog.created`;
- отдельное сохранение lag-метрик в URL, localStorage или Jira properties;
- изменение repo pipeline или status pipeline.

## Риски

- Если Jira не вернёт `worklog.created`, lag для такого worklog не должен выдумываться: запись остаётся без warning и без вклада в сумму.
- Формула специально считает календарное, а не рабочее отставание, поэтому выходные тоже попадут в результат.
- Поздний worklog за старый день попадёт в недельную мини-таблицу недели `workedDay`, а не недели `loggedAt`. Это соответствует смыслу User Activity: показывать опоздавшее списание рядом с днём работы.

## Проверка

- В worklog-строке видно `за DD.MM`.
- Если worklog внесён не в тот же день, видно `внесено DD.MM HH:MM` и `отставание Xд Yч`.
- Поздние worklog подсвечены.
- В верхней пользовательской таблице появилась колонка `Отставание`.
- В календаре у каждой недели есть мини-таблица с суммой отставания по пользователям.
- Calendar user filter пересчитывает недельную мини-таблицу вместе с остальным week render.
