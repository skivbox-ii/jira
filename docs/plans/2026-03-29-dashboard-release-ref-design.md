# Dashboard `releaseRef` rollout — design

## Контекст и рационале

Стабильный URL `*.bootstrap.js` на `@main` решает проблему «не трогать Jira config при каждом релизе», но **одного** вшитого в bootstrap build-time SHA недостаточно для управляемого rollout: все дашборды оказались бы привязаны к одному коммиту, зашитому при последней сборке генератора, без явного способа обновить версию на уровне дашборда.

Нужна модель, в которой:

- в Jira по-прежнему один неизменяемый URL на bootstrap (`@main`);
- **версия набора ассетов** (`_ujgCommon.js`, CSS, `*.runtime.js`) задаётся **на уровне дашборда** и одинакова для всех UJG-гаджетов на этом дашборде;
- первый заход может сам зафиксировать актуальный SHA с `main` без ручных правок в Jira;
- обновление до последнего `main` — явное действие пользователя, после которого следующая загрузка страницы уже использует новый закреплённый SHA.

## Архитектура (итоговая)

1. **Стабильный loader** — в настройке гаджета Jira указывается только `https://cdn.jsdelivr.net/gh/.../ujg-<widget>.bootstrap.js` с веткой/алиасом `@main` (или эквивалент). Этот файл мал и меняется редко; он не несёт «истину» о версии runtime-набора для дашборда.

2. **Общий для дашборда `releaseRef`** — строка (полный или сокращённый Git SHA), по которой строятся URL вида `.../repo@<releaseRef>/<file>`. Хранится **один раз на дашборд**, а не в каждом гаджете.

3. **Fallback в bootstrap** — при генерации в `*.bootstrap.js` остаётся `var releaseRef = "<build-time>"`: используется только если недоступны Jira Dashboard Properties и/или GitHub API, чтобы гаджет не остался без единого согласованного ref.

4. **Toolbar «Обновить версию»** — запрашивает актуальный SHA `main` с GitHub, при отличии от текущего сохраняет его в свойство дашборда и инициирует **полную перезагрузку страницы**.

## Хранилище: ключ и API

- **Ключ свойства сущности дашборда Jira:** `ujg.dashboardReleaseRef`.
- **Чтение:** REST `GET /rest/api/2/dashboard/{id}/properties/ujg.dashboardReleaseRef` (или эквивалент через Connect/JavaScript API `getDashboardProperty`, если доступен).
- **Запись:** `PUT` с телом `{"value":"<sha>"}` и заголовком `X-Atlassian-Token: no-check` (или `setDashboardProperty` в API гаджета).

Идентификатор дашборда определяется из `window.location` (`selectPageId`) и/или `AJS.params` (`selectPageId`, `pageId`).

## Поток первой загрузки

1. Jira загружает bootstrap с `@main` (AMD-модуль гаджета).
2. Bootstrap читает `ujg.dashboardReleaseRef` для текущего дашборда.
3. Если свойства нет (404 / пусто):
   - запрашивается `GET https://api.github.com/repos/<org>/<repo>/commits/main`;
   - из ответа берётся `sha`;
   - выполняется сохранение в dashboard property; при ошибке сохранения ref всё равно используется для текущей сессии (лучше показать виджет, чем блокировать из-за прав на property).
4. Если чтение свойства или внешний запрос недоступны — используется вшитый `releaseRef` из bootstrap.
5. По выбранному ref строятся URL для `_ujgCommon.js`, widget CSS и `*.runtime.js`; загрузка дедуплицируется через `window.__UJG_BOOTSTRAP__`.
6. После загрузки вызывается `require(runtimeAmd)` и создаётся экземпляр runtime-гаджета. В тулбаре отображается краткая метка версии.

## Поток обновления версии («Обновить версию»)

1. Повторно разрешается текущий ref (кэш страницы / promise в `__UJG_BOOTSTRAP__`).
2. Запрашивается свежий SHA с GitHub `main`.
3. Если SHA совпадает с текущим — только обновляется отображение в тулбаре, перезагрузки нет.
4. Если отличается — `PUT` нового значения в `ujg.dashboardReleaseRef`, обновление отображаемой версии, затем `window.location.reload()`.

После перезагрузки все гаджеты на дашборде читают уже обновлённое свойство и грузят согласованный набор файлов с нового ref.

## Обработка ошибок

- **GitHub API:** при ошибке на первой инициализации без сохранённого property используется вшитый `releaseRef`; в консоль пишется ошибка bootstrap при сбое сети/HTTP на этапе refresh.
- **Jira GET property:** при сетевой/HTTP ошибке — откат на вшитый `releaseRef` для загрузки ассетов.
- **Jira PUT property:** при ошибке сохранения на refresh — лог в консоль; перезагрузка не обязана маскировать проблему, но UX зависит от реализации (кнопка может завершиться с ошибкой без reload).
- **Загрузка script/style:** как и раньше, `onerror` + отклонение Promise и сообщение в консоль.

## Почему полная перезагрузка страницы, а не in-page AMD hot-swap

- **Состояние AMD/require:** runtime уже зарегистрирован под фиксированным именем модуля; «подменить» bundle без выгрузки предыдущего определения в среде Jira ненадёжно и зависит от внутренностей загрузчика.
- **Несколько гаджетов:** на одном дашборде несколько экземпляров и виджетов делят глобальный кэш скриптов/стилей; точечная замена одного модуля не гарантирует согласованность всех виджетов и общих ассетов (`_ujgCommon.js`).
- **CSS и побочные эффекты:** смена версии затрагивает и стили; перезагрузка гарантирует чистый DOM и отсутствие «хвостов» от старой версии.
- **Простота инварианта:** после reload все экземпляры заново проходят один и тот же путь «прочитать property → закрепить URL → загрузить».

Отказ от ранее обсуждавшегося варианта **inline hot-reload** (повторная подгрузка runtime без reload) осознан: он уменьшал бы задержку, но увеличивал бы риск рассинхрона, дублирования модулей и сложность отладки в production.
