# Multi-User Activity Dashboard — Design

Дата: 2026-03-31

## Цель

Расширить дашборд активности (`ujg-user-activity-modules`) для работы с несколькими пользователями: мультиселект, объединённый календарь Jira+Repo, таймстемпы и авторы на всех действиях, подсветка отсутствия трудозатрат.

## Изменяемые модули

| Модуль | Действие |
|--------|----------|
| `user-picker.js` | Заменяется на `multi-user-picker.js` |
| `calendar-heatmap.js` | Объединяется с `repo-calendar.js` → `unified-calendar.js` |
| `repo-calendar.js` | Удаляется (логика переносится в unified-calendar) |
| `api.js` | Добавляется `fetchIssueComments()` |
| `data-processor.js` | Мультипользовательская структура `dayMap` |

| `rendering.js` | Оркестрация параллельной загрузки, новая интеграция |
| `repo-data-processor.js` | Фильтрация по нескольким пользователям |
| `summary-cards.js` | Разбивка статистики по пользователям |
| `activity-log.js` | Реальные таймстемпы вместо Math.random() |
| `config.js` | Новые цвета/стили для чипов пользователей |

## 1. Мультиселект пользователей

Новый модуль `multi-user-picker.js` заменяет `user-picker.js`.

**UI:**
- AUI-кнопка с адаптивной подписью: «0 пользователей» / «Иванов И.» / «3 выбрано»
- Выпадающая панель:
  - Поле поиска → `GET /rest/api/2/user/picker?query=...`
  - Результаты с чекбоксами
  - Блок «Выбранные» — чипы с крестиком
  - Кнопка «Сбросить»
- Закрытие по клику снаружи

**Состояние:**
```
selectedUsers: [{ name, displayName, key }]
searchResults: [...]
searchQuery: ""
```

**URL:** параметр `users=ivanov,petrov` (через запятую).

**Интеграция:** `onChange(selectedUsers)` → `rendering.js`. Кнопка «Загрузить» запускает `loadData(selectedUsers, period)`.

## 2. Загрузка и обработка данных

**Подход:** параллельная загрузка по пользователям.

Для каждого пользователя из `selectedUsers` параллельно:
1. `api.fetchAllData(user.name, start, end, onProgress)` — ворклоги + changelog
2. `api.fetchIssueComments(issueKeys, user.name)` — комментарии к задачам (новый вызов, `GET /rest/api/2/issue/{key}/comment`)
3. `repoApi.fetchRepoActivityForIssues(issues, ...)` — один вызов на все задачи

**Прогресс:** «Загрузка: Иванов (3/10), Петров (7/15)...»

**Новая структура `dayMap[dateKey]`:**
```
{
  users: {
    "ivanov": { worklogs, changes, comments, totalHours },
    "petrov": { worklogs, changes, comments, totalHours }
  },
  allWorklogs: [...],     // все ворклоги с полем author
  allChanges: [...],      // changelog с полем author
  allComments: [...],     // комментарии к задачам с полем author
  totalHours: <сумма>,
  repoItems: [...]        // коммиты/PR — уже имеют автора
}
```

Каждый элемент содержит:
- `author: { name, displayName }`
- `timestamp` — реальное время из API

`issueMap` — общий, дедупликация по ключу задачи.

## 3. Объединённый календарь

Один модуль `unified-calendar.js` вместо `calendar-heatmap.js` + `repo-calendar.js`.

**Структура ячейки:**
```
┌─────────────────────────────┐
│ Пн 15 мар                   │  заголовок
├─────────────────────────────┤
│ [Иванов 4.5ч] [Петров 0ч🔴]│  чипы пользователей
├─────────────────────────────┤
│ JIRA-123 Иванов 2ч 09:30   │  верхний блок:
│ JIRA-456 Петров 💬 11:00   │  ворклоги + комменты + changelog
│ JIRA-789 → In Progress 14:00│  сортировка по timestamp
├─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┤  граница (выровнена по неделе)
│ 🟢 commit abc123 16:20     │  нижний блок:
│ 🔵 PR #42 merged 17:00     │  repo-активность
└─────────────────────────────┘
```

**Выравнивание границы по неделе (двухпроходный рендер):**
1. Рендер верхнего блока каждого дня в неделе
2. `maxJiraHeight = Math.max(...jiraBlockHeights)` по всем дням строки
3. `min-height: maxJiraHeight` на верхний блок всех дней в строке

Используется `requestAnimationFrame` после вставки DOM для измерения → применения min-height.

**Содержимое верхнего блока:**
- Ворклоги: `[ключ] [автор] [часы] [HH:MM]`
- Комментарии: `[ключ] [автор] 💬 [HH:MM]`
- Changelog: `[ключ] [автор] → [статус] [HH:MM]`
- Всё отсортировано по timestamp

**Содержимое нижнего блока:**
- Коммиты, PR, ветки — с автором и временем
- Сортировка по timestamp

**Цвет ячейки:** по суммарным часам всех пользователей через `getHeatBg()`.

## 4. Детализация дня

Доработка `daily-detail.js`.

Все действия сгруппированы по задачам, внутри — хронологически:
```
┌─ Понедельник, 15 марта 2026 ──────────── [✕] ─┐
│ ┌─ JIRA-123: Доработка API ─────────────────┐  │
│ │ 09:30 Иванов И. — Worklog 2ч              │  │
│ │       "Реализовал эндпоинт /users"        │  │
│ │ 11:15 Петров А. — Комментарий             │  │
│ │       "Нужно добавить валидацию"           │  │
│ │ 14:00 Иванов И. — Open → In Progress      │  │
│ └───────────────────────────────────────────┘  │
│ ┌─ JIRA-456: Баг в авторизации ─────────────┐  │
│ │ 10:00 Петров А. — Worklog 1.5ч            │  │
│ │ 16:20 Петров А. — commit abc1234          │  │
│ └───────────────────────────────────────────┘  │
│ ── Без привязки к задаче ──                    │
│ 17:00 Иванов И. — PR #42 merged               │
└────────────────────────────────────────────────┘
```

Каждое действие: **HH:MM**, **автор (ФИО)**, **тип**, **детали**.
Реальные таймстемпы из API вместо Math.random().
Репо-события привязываются к задачам через issue key из dev-status.

## 5. Красная подсветка отсутствия трудозатрат

Для каждого рабочего дня (пн-пт), для каждого выбранного пользователя: `users[userId].totalHours === 0` → красный чип.

Правила:
- Только рабочие дни (пн-пт)
- Только прошедшие и текущий день (не будущие)
- Все пользователи без часов → красный бордер ячейки
- Часть без часов → только их чипы красные
- Выходные не подсвечиваются

В summary cards: «Дней без трудозатрат: Иванов — 2, Петров — 5».
