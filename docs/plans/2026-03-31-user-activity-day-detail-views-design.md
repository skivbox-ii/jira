# User Activity Day Detail Views Design

Дата: 2026-03-31

## Цель

Расширить `user-activity` так, чтобы:
- везде, где показывается `issueKey`, он был ссылкой на Jira issue в новой вкладке;
- название задачи в детализации дня всегда показывалось полностью, даже если блок станет выше;
- детализация дня поддерживала два режима: текущий `По задачам` и новый `По команде`;
- у repo-событий явно отображались автор коммита и текущий Jira status задачи, если задача известна.

## Ключевые решения

### 1. Два режима детализации дня

Детализация дня остаётся одной точкой входа по клику на ячейку календаря, но внутри получает переключатель:
- `По задачам` — режим по умолчанию, основан на текущем UI;
- `По команде` — новый режим с колонками пользователей и общей вертикальной временной шкалой.

Оба режима работают от одного и того же нормализованного списка действий дня. Меняется только раскладка.

### 2. Режим `По задачам`

Этот режим сохраняет текущую модель чтения:
- действия сгруппированы по `issueKey`;
- внутри группы все действия отсортированы по `timestamp`;
- действия без `issueKey` попадают в блок `Без привязки к задаче`.

Улучшения:
- `issueKey` всегда рендерится как ссылка на Jira issue с `target="_blank"`;
- `summary` задачи никогда не режется `truncate`, а переносится на новые строки;
- repo-события внутри группы показывают автора и текущий Jira status задачи;
- если задача неизвестна, показываем только те поля, которые реально есть у события.

### 3. Режим `По команде`

Новый режим показывает, как проходил день команды во времени:
- слева идёт общая временная шкала;
- справа идут вертикальные колонки пользователей;
- действие попадает в колонку своего автора;
- положение блока по `Y` определяется временем начала действия;
- шкала времени мягкая, без жёстких часовых строк.

За базовый вертикальный диапазон берётся фактическое окно дня:
- от самого раннего действия;
- до самого позднего действия;
- с небольшими верхним и нижним отступами.

Это даёт более полезный вид для командного дня, чем сетка `00:00..23:59`, где большая часть пространства пустая.

### 4. Коллизии действий в таймлайне

Если у одного пользователя несколько действий почти в одно время:
- не делаем сложный layout engine с раздвижением карточек;
- складываем их стопкой в близкой вертикальной зоне;
- порядок внутри зоны остаётся хронологическим.

Решение специально упрощённое: оно устойчивее и лучше соответствует плотности данных в gadget UI.

### 5. Действия без точного времени

Если действие не имеет валидного `timestamp`:
- оно не попадает в основной таймлайн;
- вместо этого рендерится в отдельном нижнем блоке `Без точного времени`.

Это исключает ложную визуальную точность и сохраняет событие в интерфейсе.

## Единые правила рендера

### Ссылки на задачи

Добавляется общий helper, который по `issueKey` строит Jira URL вида:

`<jira-base-url>/browse/KEY`

Правило единое для:
- `daily-detail`;
- строк Jira в `unified-calendar`;
- строк repo-активности в `unified-calendar`;
- логов, если там выводится `issueKey`.

### Название задачи

Для day-detail summary задачи:
- всегда показывается полностью;
- переносится строками;
- не скрывается ради компактности.

Это важнее экономии вертикального места, потому что основной сценарий тут именно чтение деталей дня.

### Repo-события

Для repo-событий всегда пытаемся показать:
- время;
- автора;
- `issueKey` как ссылку, если он есть;
- текущий Jira status задачи, если задача найдена в `issueMap`;
- commit message / PR title / branch label как вторичный текст.

Если `issueKey` отсутствует:
- не рисуем ссылку;
- не рисуем status badge;
- событие остаётся repo-only.

## Источник данных для статуса задачи

Текущий Jira status для commit/PR/branch события берётся из уже построенного `issueMap`:
- `repoItem.issueKey` уже присутствует в repo-данных;
- `issueMap[issueKey].status` уже заполнен после основной Jira загрузки.

Дополнительный запрос на каждый commit не нужен.

Это решение намеренно использует именно текущий status задачи на момент открытия дашборда, а не исторический status на момент коммита.

## Нормализованная модель действия дня

Перед рендером оба режима используют единый action list с полями:

```js
{
  type: "worklog" | "change" | "comment" | "repo",
  subtype: "commit" | "pull_request_opened" | ...,
  timestamp: "...",
  author: { name, displayName },
  issueKey: "ABC-123",
  issueSummary: "Summary",
  issueStatus: "In Progress",
  text: "...",
  comment: "...",
  message: "...",
  linkUrl: "https://jira.example.com/browse/ABC-123"
}
```

Далее:
- режим `По задачам` группирует этот список по `issueKey`;
- режим `По команде` группирует по автору и раскладывает по времени.

## Out of Scope

В эту задачу не входят:
- вычисление исторического статуса задачи на момент коммита;
- новый API-слой для repo-событий;
- строгая почасовая сетка;
- сложный layout engine с разрешением визуальных пересечений;
- переделка общей модели загрузки dashboard.

## Обработка ошибок

- Нет `issueKey` -> нет ссылки и нет status badge.
- Нет `summary` -> показываем только ключ и тип действия.
- Нет автора у repo-события -> fallback `Неизвестно`.
- Нет валидного времени -> событие уходит в блок `Без точного времени`.
- Очень длинный текст -> перенос строк, без принудительного обрезания в day-detail.

## Проверка

Нужно покрыть:
- `issueKey` рендерится ссылкой с `target="_blank"`;
- summary в day-detail не режется;
- repo-событие показывает автора и текущий Jira status;
- детализация дня переключается между `По задачам` и `По команде`;
- режим `По команде` строит колонки пользователей и общую временную шкалу;
- действия без времени попадают в отдельный нижний блок;
- существующий режим `По задачам` не теряет текущую детализацию по issue.
