# Story Browser epic-first hierarchy — design

## Контекст

Текущий `ujg-story-browser` строит дерево из общего project-wide списка задач и потому расходится с ожидаемым UX:

- в корнях оказываются лишние сущности;
- фильтр по эпикам неудобен для выбора;
- ключи задач не кликабельны;
- структура `Epic -> Story -> child tasks` теряется;
- таблица визуально не соответствует референсу `https://preview--daily2-diligence-board.lovable.app/stories`.

Нужен переход на **epic-first pipeline** с сохранением текущих режимов представления (`Таблица`, `Аккордеон`, `Строки`), но с новой моделью данных и новым фильтром эпиков.

## Цели

1. В фильтре показывать **все** эпики проекта: открытые и закрытые.
2. Фильтр эпиков сделать **searchable multi-select** с подписью `KEY - summary`, сортировкой по номеру задачи и выбором нескольких эпиков сразу.
3. В основном дереве показывать корнями только **открытые** эпики.
4. Под каждым эпиком показывать только задачи типа **`Story`**.
5. Под каждой историей показывать только прямых детей из `issuelinks` с типами связи `child` / `is_child`.
6. Ключ каждой задачи рендерить как ссылку `JIRA_BASE_URL/browse/KEY`.
7. Для child-задач вычислять классификацию по префиксу в summary (`[BE]`, `[FE]`, ...); отсутствие префикса подсвечивать красным.
8. Привести заголовок и таблицу максимально близко к референсу `/stories`.

## Не-цели

- Не возвращать старую project-wide иерархию `parent -> epicLink` как источник истины для Story Browser.
- Не показывать под эпиком прямые не-`Story` задачи.
- Не тащить произвольных потомков глубже одного уровня от `Story`.

## Источник истины по данным

### 1. Каталог эпиков для фильтра

При выборе проекта сначала загружается каталог эпиков проекта:

- в фильтре доступны **все** эпики, независимо от статуса;
- список сортируется по числовой части ключа (`ABC-2`, `ABC-10`, `ABC-100`);
- опция отображается как `KEY - summary`;
- выбранных эпиков может быть несколько;
- состояние хранится как массив `selectedEpicKeys`.

Если пользователь не выбрал эпики явно, виджет работает как `all epics selected`, но в рендере корнями всё равно остаются только открытые.

### 2. Корни дерева

После определения набора выбранных эпиков виджет строит display-tree только по **открытым** эпикам:

- закрытые эпики остаются доступны в фильтре;
- закрытые эпики не становятся корнями таблицы/аккордеона/строк;
- если пользователь выбрал только закрытые эпики, результатом будет пустое состояние без искусственного fallback.

### 3. Истории внутри эпиков

Для выбранных открытых эпиков подгружаются только задачи типа `Story`.

Прямые `Task`, `Bug`, `Sub-task` и прочие задачи под эпиком не попадают в display-tree.

### 4. Дети историй

После загрузки историй из их `issuelinks` извлекаются прямые дочерние ключи только по связям:

- `child`
- `is_child`

Далее эти задачи догружаются отдельным запросом по `key in (...)`.

Итоговая структура данных:

`Epic -> Story -> Child Task`

Глубже одного уровня от `Story` дерево не строится.

## Нормализация и классификация

Для каждого узла сохраняются как минимум:

- `key`
- `summary`
- `status`
- `statusCategory`
- `type`
- `sprint`
- `labels`
- `components`
- `isDone`
- `browseUrl`

Для child-задач дополнительно вычисляется:

- `classification`: значение из префикса summary, например `BE`, `FE`, `QA`, `DEVOPS`
- `classificationMissing`: `true`, если префикс не найден

Правило классификации:

- summary начинается с `[\w+]` -> берём содержимое внутри скобок;
- если префикса нет, показываем fallback `NO PREFIX` и красное состояние.

## Фильтры и состояние UI

Заголовок должен остаться близким к референсу и содержать:

- `Проект` — searchable single select;
- `Статус` — searchable single select / all;
- `Эпик` — searchable multi-select с chips;
- `Спринт` — searchable single select / all;
- `Поиск` — полнотекстовый фильтр;
- кнопки режимов `Таблица`, `Аккордеон`, `Строки`;
- `Развернуть всё` / `Свернуть всё`.

Фильтры `Статус`, `Спринт`, `Поиск` применяются **после** построения иерархии `Epic -> Story -> Child Task`.

Сохраняемое состояние:

- проект;
- массив выбранных эпиков;
- статус;
- спринт;
- поисковая строка;
- режим представления;
- набор раскрытых ключей.

Старый сохранённый single-value epic filter должен мягко мигрировать в массив из одного элемента.

## Таблица

Колонки:

- `Классификация`
- `Ключ`
- `Название`
- `Статус`
- `Спринт`
- `Метки`
- `Компоненты`

Правила отображения:

- `Epic` строки: в `Классификация` ставим `EPIC`, в `Метки` и `Компоненты` можно показывать `-` при отсутствии значения;
- `Story` строки: классификация `STORY`;
- child-строки: классификация из summary prefix, либо красный `NO PREFIX`;
- `Ключ` всегда ссылка на Jira;
- `Метки` и `Компоненты` отображаются компактно, через запятую, с обрезкой при длинных списках;
- визуальная иерархия строится отступами и/или маркерами типа.

## Аккордеон и строки

Оба режима используют **тот же самый tree-model**, что и таблица.

Различается только представление:

- `Аккордеон`: фокус на раскрытии `Epic -> Story`;
- `Строки`: компактные карточки/ряды;
- `Ключ` остаётся ссылкой;
- `Классификация`, `Метки`, `Компоненты` не теряются, а показываются в компактном виде.

## Ошибки и пустые состояния

- Нет выбранного проекта: показываем штатное пустое состояние.
- Выбраны только закрытые эпики: пустое состояние без падения.
- Нет `Story` в эпиках: пустое состояние с нормальным header/filter state.
- Нет child-задач у истории: история рендерится без дочернего списка.
- Ошибка любого Jira запроса: компактный error-state в контентной области, header остаётся доступным.

## Производительность

Вместо одного огромного project-wide search используем staged loading:

1. загрузка эпиков проекта;
2. загрузка `Story` только для нужных открытых эпиков;
3. догрузка child issues только по реально найденным linked keys.

Это уменьшает шум в памяти и убирает лишние закрытые/нерелевантные задачи из основного дерева.

## Тестовая стратегия

Нужны регрессии для:

- каталога эпиков: все эпики, сортировка по номеру, multi-select state;
- display roots: только открытые эпики;
- под эпиком только `Story`;
- child-слой только из `child` / `is_child` links;
- классификация `[BE]`, `[FE]`, `NO PREFIX`;
- кликабельные ключи;
- таблица/аккордеон/строки используют одну и ту же иерархию;
- фильтры `Статус`, `Спринт`, `Поиск` работают поверх новой модели.
