# User Activity Render-Time User Filter — Design

Дата: 2026-04-01

## Цель

Вернуть явный фильтр по пользователю в календарь и в детализацию дня, не откатывая `hard-open`, не меняя pipeline и не трогая текущую `repo stream` модель.

## Контракт

- `repo-data-processor.js`, `data-processor.js` и `rendering.js` не меняют состав данных ради фильтра.
- Repo-активность остаётся отдельным потоком в `dayMap[date].repoItems`.
- Фильтрация выполняется только на этапе рендера.
- Календарь и `day detail` хранят фильтр независимо друг от друга.
- Фильтр работает как `multi-select`.
- Опции фильтра берутся только из `selectedUsers`, выбранных в самом дашборде.
- По умолчанию все выбранные пользователи включены.

## Проблема

`Hard-open` убрал исчезновение repo-событий и позволил видеть полный поток авторов. После этого пропал способ быстро сузить картину до нужных людей. Возвращать старую downstream-фильтрацию нельзя: она снова смешает загрузку данных, author matching и layout.

Нужен простой слой поверх уже загруженных данных: показывать или скрывать записи при рендере, но не вмешиваться в то, что пришло из Jira и dev-status.

## Решение

Фильтр добавляется в два UI-узла:

- `unified-calendar.js`
- `daily-detail.js`

Оба модуля получают локальный state вида «какие индексы из `selectedUsers` сейчас активны». Данные в `dayMap`, `issueMap` и нормализованных repo items не меняются. Перед выводом UI модуль строит временную `visible`-модель и рендерит уже её.

## Общее правило фильтрации

Для обоих экранов действует один и тот же контракт:

1. Если активны все пользователи из `selectedUsers`, UI показывает исходные данные без сужения.
2. Если активна только часть пользователей, UI показывает только те записи, чей автор совпал хотя бы с одним активным пользователем.
3. Если ни один пользователь не активен, UI честно показывает пустой результат.

Это правило важно для `hard-open`: авторы вне набора `selectedUsers` видны только в режиме `все включены`. Как только пользователь сужает выбор, остаются только записи, совпавшие с активными dashboard users.

## Сопоставление автора

Сопоставление автора остаётся мягким и выполняется только в UI-слое. Для сравнения используются те же поля, которые уже встречаются в коде и тестах:

- `accountId`
- `key`
- `name`
- `userName`
- `displayName`

Нужен один общий helper в `utils.js`, чтобы календарь и `day detail` не разошлись по правилам author match.

## Календарь

### UI

Над таблицей календаря появляется компактная строка toggle-кнопок по `selectedUsers`. Каждая кнопка включает или выключает одного пользователя. State живёт внутри экземпляра `unifiedCalendar.render(...)`.
В текущей реализации кнопки используют атрибут `data-ua-cal-user-idx`, а общие стили панели переиспользуют классы `.ujg-ua-cal-user-filter-bar` и `.ujg-ua-cal-user-filter*`.

### Что именно фильтруется

Календарь строит временную `visibleDayMap` из текущего `dayMap`:

- `allWorklogs`, `allChanges`, `allComments`, `repoItems` сужаются по активным пользователям;
- `users` сужается до активных dashboard users;
- `totalHours` пересчитывается только по видимым worklog;
- column totals и week totals пересчитываются из `visibleDayMap`, а не из исходного `dayMap`.

За счёт этого фильтр влияет не только на строки Jira/repo, но и на суммы в ячейках, неделях и столбцах.

### Что не меняется

- исходный `dayMap`;
- merge repo items в `rendering.js`;
- логика выбора дня;
- существующий `hard-open` поток данных.

## Детализация дня

### UI

В `daily-detail.js` рядом с текущим переключателем режима (`По задачам / По команде`) появляется независимая строка toggle-кнопок по тем же `selectedUsers`.
В текущей реализации кнопки используют атрибут `data-ua-detail-user-idx` и те же классы панели фильтра, что и календарь.

State живёт внутри экземпляра `dailyDetail.create()` и не синхронизируется с календарём.

### Что именно фильтруется

`renderInner(...)` сначала строит `normalized = normalizeDayActions(dayData, issueMap)`, затем создаёт `visibleActions` только для текущего рендера:

- `issue view` группирует только `visibleActions`;
- `team view` строит timeline только по `visibleActions`;
- repo-таблицы за день используют только `visibleRepoActions`.

Колонки team view по-прежнему строятся по фактическим авторам оставшихся действий, а не по полному `selectedUsers`.

## Out of Scope

В эту задачу не входят:

- изменение fetch pipeline;
- возврат дедупликации в `repo-data-processor.js`;
- фильтрация summary cards, project breakdown, issue list, activity log, repo log;
- сохранение фильтра в URL, localStorage или между reload;
- добавление новых источников repo-данных.

## Риски

- Если выключить всех пользователей, календарь и day detail покажут пустой результат. Это ожидаемое поведение `multi-select`.
- При частичном выборе исчезнут repo-события авторов вне `selectedUsers`, хотя в режиме `все включены` они видны. Это осознанная часть контракта.
- Календарю придётся пересчитывать временную `visibleDayMap` на каждом repaint, но объём данных уже ограничен текущим окном периода и числом выбранных пользователей.

## Проверка

- В календаре появляются toggle-кнопки по выбранным dashboard users.
- При выключении пользователя из календаря пропадают его Jira и repo записи, а day totals и week totals пересчитываются.
- При включённых всех пользователях календарь продолжает показывать `hard-open` поток, включая repo-авторов вне `selectedUsers`.
- В `day detail` фильтр работает независимо от календаря.
- В `team view` остаются только колонки авторов из видимых действий.
- Pipeline загрузки и merge repo items не меняются.
